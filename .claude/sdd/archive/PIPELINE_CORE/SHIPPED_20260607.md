## Lessons Learned

### Process

**Lesson 1: External API specs must be verified against live systems before design locks in code patterns**

**What happened:** The DESIGN documented a non-existent BACEN endpoint `CotacaoTodosComercialDia` that would return all 19 currencies in one call. The design was written against hallucinated API behavior. When the pipeline ran against real BACEN OData metadata, this endpoint did not exist — the actual API uses individual `CotacaoMoedaDia(moeda, dataCotacao)` function calls per currency. The code pattern had to be rewritten mid-build.

**Why it matters:** Unit tests mock external APIs, so hallucinated specs pass all tests. Only real end-to-end execution against live systems reveals these gaps. The pipeline appeared 100% "complete" with passing unit tests but was entirely broken in production.

**Action:** Before /design locks in code patterns for external integrations, schedule a 30-minute spike to hit the real API endpoint, inspect response shape, and document actual behavior (not intended behavior). For this project: verified BACEN PTAX endpoint exists via `curl https://olinda.bcb.gov.br/olinda/servico/PTAX/versao/v1/odata/$metadata`, confirmed `CotacaoMoedaDia` function and `CotacaoTodosComercialDia` does not exist.

---

**Lesson 2: Nested archive formats (ZIP inside ZIP/SFX) are not handled by naive zipfile.extractall**

**What happened:** B3's `.ex_` file is not a simple ZIP — it's a ZIP containing a self-extracting archive (PKSFX/.exe) with the same filename as the outer archive. Calling `zipfile.extractall()` into the same directory overwrote the source file mid-read, causing `EOFError` and truncating the `.ex_` to 0 bytes.

**Why it matters:** The file format was assumed to be a simple ZIP based on documentation. Real B3 downloads revealed a nested structure. This was a silent failure (HTTP 200, file exists, but truncated), only visible in production.

**Action:** Added handling to read both archive layers fully into memory using `io.BytesIO` before extraction, and validation to detect empty/truncated downloads (B3 server intermittently returns 200 OK with 0 bytes). Retry on empty response.

---

**Lesson 3: "Divisor" fields baked into config become invisible bugs when the numeric field is already fully decoded**

**What happened:** The B3 numeric field uses an embedded decimal indicator (last 2 digits) to define scale. The `_decode_valor()` function fully decodes this to the final spot rate (e.g. `+00000000000000000005078504` → `5.0785`). The PRD listed a "divisor extra" table (÷1.000 for most, ÷10.000.000 for EUR), which was baked into the `Instrumento` dataclass. When this divisor was applied on top of the already-decoded value, every rate became nonsense (USD/BRL = 0.0050785 instead of 5.0785).

**Why it matters:** The "divisor extra" was either hallucinated (not real) or misunderstood (already applied in decodification). It corrupted every single value by 3–4 orders of magnitude. Unit tests passed because they used small mocked numbers. Real B3 data (19 currencies × 1000+ lines) revealed the error immediately in e2e.

**Action:** Removed the `divisor` field entirely from `Instrumento` config. The numeric field decode is the single source of truth for scale. Verified against real B3 values: USD/BRL ≈ 5.15 (correct), EUR/BRL ≈ 5.93, JPY/BRL ≈ 0.032 — all sane real-world market rates.

---

### Technical

**Lesson 4: Suffix-matching logic must match the FULL configured suffix, not truncate at delimiters**

**What happened:** The instrument search function `_buscar_instrumento` used `sufixo.split("-")[0]` to search, so config suffix `"DOL-D1"` was truncated to `"DOL"`, and `"DOL-D1".endswith("DOL")` evaluated to `False`. Almost no currencies matched real B3 instruments.

**Why it matters:** The business rule ("search by suffix, not exact match") was misinterpreted in code. The real B3 file has instruments like `"RTDOL-D1"`, `"RTREUR-D1"`, `"RTGBP-PF"` — the suffix IS the full configured name. Naive truncation breaks the match.

**Action:** Fixed to match on the full configured suffix. Verified against real Indic.txt that all 19 instruments now match correctly.

---

**Lesson 5: BACEN holiday API endpoint documented in PRD also does not exist**

**What happened:** The DESIGN referenced a BACEN OData endpoint `Feriados` to fetch Brazilian holidays. This endpoint does not exist in the real PTAX OData service. The fix was to switch to `BrasilAPI` (`brasilapi.com.br/api/feriados/v1/{ano}`), a real public Brazilian holiday API, and update its mocked response shape in tests.

**Why it matters:** Same root cause as lesson 1 — external specs were not validated. The calendar module was completely broken against reality until the correct API was found and plugged in.

**Action:** Updated `src/calendario.py` to use BrasilAPI instead. Verified the API returns correct holidays for 2026 via manual request.

---

### Communication

**Lesson 6: End-to-end testing against live external systems should be a Phase 3 (BUILD) checklist item before /ship**

**What happened:** The build was marked "complete" with 17/17 unit tests passing. Only after /ship was initiated did we run the pipeline against real 2026-06-05 B3 and BACEN data. Five critical bugs surfaced instantly.

**Why it matters:** Mocked external systems hide all integration bugs. The project was technically "done" (tests pass, code compiles) but not actually done (completely broken in production).

**Action:** Added a required checklist step at the end of /build: "Run end-to-end execution against real external systems for 1 recent date; validate all 19 currencies are present with sane values (no 0s, no >100% spreads)." This applies especially to pipelines that depend on external APIs.

## Key Takeaways

1. **Mock tests are necessary but not sufficient.** They validate code logic but hide integration bugs. For systems with external dependencies, allocate time for real e2e testing in Phase 3.

2. **API documentation is often outdated or inaccurate.** Always verify endpoint existence, response schema, and field values against the real system before locking code patterns into DESIGN.

3. **Nested archive formats and binary downloads need careful handling.** Read into memory when possible; validate checksum or content before trusting the file.

4. **Numeric field encoding must be fully understood before applying further transformations.** The "divisor extra" was a red herring — the B3 field already includes all scaling info.

5. **Suffix matching is more fragile than expected.** The business rule ("search by suffix") required matching the full configured suffix, not truncating at delimiters.
